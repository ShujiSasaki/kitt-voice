判定方式の変更

変更前（係数方式）変更後（AI直接判定）
7項目の重み付けスコア計算BQ全蓄積データをGeminiに渡して直接判断
score ≥ 0.85 → acceptGeminiが文脈（クエスト・天気・疲労・エリア需要）を総合判断
dynamic_coefficientsテーブルに依存テーブル廃止。データが溜まるほどAIが賢くなる

文脈の永続化

セッション1 (0:00-0:15)
  オファー3件蹴った、1件受けた → BQに全部記録

セッション2 (0:15-0:30) ← 自動再接続
  ① ContextBridgeClient → CF /contextBridge GET
  ② CF が BQ から並列5クエリ (3秒以内)
  ③ 今日の全履歴+会話サマリーを systemPrompt に注入
  → KITTは「さっき3件蹴って1件受けた」を知ってる状態で再開

短期記憶 = Live Audioセッション内(15分) / 長期記憶 = BQ(永遠)

受諾後フロー（状態遷移）

IDLE → OFFER_DETECTED → OFFER_JUDGING → OFFER_REPORTED
  ↑                                         ↓
  │    ←←← reject ←←←←←←←←←←←←←←←←←←←←←
  │                                         ↓ accept
  │    ACCEPTED → PICKING → AT_PICKUP → DELIVERING → DROPPED → IDLE
  │        ↓          ↓          ↓           ↓           ↓
  │      GPS+BQ     GPS+BQ    GPS+BQ      GPS+BQ     GPS+BQ+リザルト
  └────────────────────────────────────────────────────────┘
ステップ検知方法アクション
OFFER_DETECTEDA11yがオファーUI検知CF判定送信
ACCEPTEDaccept_offer function callingピック先ナビ→iPhone
AT_PICKUPGPS店舗座標50m以内 or A11y「ピック完了」配達時間計測開始
DELIVERINGA11yピック完了後の画面変化ドロップ先ナビ→iPhone
DROPPEDA11y「配達完了」テキストリザルト→BQ自動保存
